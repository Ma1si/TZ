# Система разграничения прав доступа (RBAC)

Реализована ролевая модель контроля доступа, основанная на трёх сущностях.

## 1. Users — пользователи

Атрибуты:
- `id` — уникальный идентификатор пользователя.  
- `first_name` — фамилия пользователя.  
- `last_name` — имя пользователя.  
- `surname` — отчество пользователя.  
- `email` — электронная почта пользователя.  
- `hash_password` — зашифрованный пароль пользователя.  
- `created_at` — время добавления записи в БД.  
- `is_active` — флаг состояния аккаунта (`true` — аккаунт рабочий, `false` — аккаунт логически удалён).

## 2. Roles — роли

Атрибуты:
- `id` — уникальный идентификатор роли.  
- `name` — название роли (например, `admin`, `user`, `manager`).  
- `description` — описание роли.

## 3. Permissions — разрешения

Атрибуты:
- `id` — уникальный идентификатор разрешения.  
- `name` — уникальное имя права (например, `users:read`, `admin:manage`).  
- `description` — описание разрешения.  
- `resource` — название ресурса, к которому относится право (например, `users`, `orders`, `admin`).

---

## Связи между сущностями

### Пользователи и роли

Между `Users` и `Roles` реализована связь многие‑ко‑многим.  
Таким образом:
- один пользователь может иметь несколько ролей;  
- одна роль может быть назначена нескольким пользователям.

### Роли и права

Между `Roles` и `Permissions` также реализована связь многие‑ко‑многим.  
Это позволяет:
- гибко связывать наборы разрешений с каждой ролью;  
- при изменении прав одной роли не затрагивать другие роли.

## Логика проверки доступа

1. Пользователь отправляет запрос к одному из защищённых роутов.
2. Из токена (JWT) извлекаются данные:
   - `sub` — идентификатор пользователя;  
   - `roles` — список ролей.
3. Если:
   - токен не валиден,  
   - `sub` не найден,  
   - `roles` отсутствуют или пусты,  
   возвращается ошибка **401 Unauthorized**.
4. Если токен валиден, роут проверяет права:
   - с помощью проверки ролей и связанных разрешений.
5. Если:
   - нужного права нет — возвращается **403 Forbidden**.  
   - нужное право есть — выполняется соответствующая операция (CRUD, чтение данных и т.п.).

## Запуск для тестирования

Для первичного тестирования запустите:

```
python init_db.py
```

Этот скрипт:
- создаёт таблицы в БД;  
- заполняет их базовыми тестовыми данными (роли, права, пользователей, связи).
